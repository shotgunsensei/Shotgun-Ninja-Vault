Integrate a new multi-tenant secure file intake module directly into TechDeck as a first-class feature, not a separate application.

This module should feel like a natural extension of TechDeck’s existing MSP / IT operations platform and should be designed for MSPs and regulated clients who need a secure way to receive files from outside users.

Module Name

Use one of these labels consistently across the UI:

Secure Upload Portal
Secure Intake
Client File Intake

Preferred internal name: secure-intake

This must live inside TechDeck’s architecture, navigation, auth model, styling system, and tenant/account system.

Product Intent

TechDeck should gain a new module that allows each TechDeck tenant to:

provision secure upload spaces for their own organization
segregate file intake by tenant and optionally by client/customer/workspace
issue temporary upload access to outside users
receive files securely from people outside their organization
track audit logs for every invite, upload, download, and permission event
enforce storage and plan limits by subscription tier
support compliance-oriented workflows for healthcare, legal, finance, and MSP use cases

This is not a separate product, separate login domain, or separate deployment.
It is a module within TechDeck with shared platform identity and shared admin control, while preserving strict tenant separation.

Multi-Tenant Requirements

TechDeck already operates in a tenant-aware business context, so this module must inherit and enforce that model.

Each TechDeck tenant must have:

isolated upload records
isolated file metadata
isolated storage namespace
isolated invites
isolated audit events
isolated policies
isolated storage limits
isolated external uploader sessions

No tenant should ever be able to access or enumerate another tenant’s intake data.

All backend queries must be tenant-scoped.

Addressing Semantics

This module must support tenant-aware addressing for external upload access without becoming a separate product.

Support these patterns:

path-based tenant routing within TechDeck
example: /t/acme/secure-intake/upload/<token>
workspace/client-based routing inside tenant context
example: /app/secure-intake/spaces/hr-onboarding
example: /app/secure-intake/spaces/medical-records
internal storage namespacing
example: tenant/acme/space/medical-records/uploads/...

Architect it so that TechDeck can later support subdomains if desired, but default to path-based routing for now.

Main Use Cases
1. Tenant staff generating upload access

A TechDeck tenant admin or authorized staff member can:

create an upload request
assign it to a client, patient, vendor, applicant, or external party
set expiration
set max upload count
set max total size
set per-file size limits
require metadata fields
restrict file types
choose one-time or reusable upload access
revoke the request at any time
2. External person uploading files

An outside user receives:

temporary credentials
or a secure expiring link
or both

They can then:

authenticate into only the specific upload area
provide required metadata
upload one or more files
receive a confirmation / receipt
not see anything beyond their authorized upload target
3. Tenant reviewing submissions

Internal TechDeck users can:

review uploaded files
filter and search by uploader/date/request/client
review metadata
view audit trails
download files if permitted
move files through workflow states
archive or soft-delete according to policy
